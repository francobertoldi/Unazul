---

## 0. Project Decision Priority (heredado de arquitectura)

1. **Estabilidad** — Aislamiento de fallos, consistencia de datos, disponibilidad.
2. **Mantenibilidad** — Ownership claro, bajo acoplamiento, testabilidad.
3. **Performance** — Tiempos de respuesta aceptables bajo carga normal.
4. **Costo** — Optimizar recursos sin comprometer los tres anteriores.

## 2. Decisiones de Brainstorming Cerradas

| ID | Decision | Opcion elegida | Razon |
|----|----------|---------------|-------|
| D-SEC-01 | Bloqueo por intentos fallidos | Bloqueo permanente tras 5 intentos | Simplicidad, estabilidad, alineado con HU038 |
| D-SEC-02 | Refresh token strategy | Rotacion estricta one-time use | Modelo ya lo soporta; detecta reuso revocando cadena |
| D-SEC-03 | Propagacion de cambios de permisos | Recargar al refresh token | Balance estabilidad/mantenibilidad; JWT corto 15-30min |
| D-CFG-01 | Invalidacion de cache al modificar parametros | Evento async ParameterUpdatedEvent via RabbitMQ | Desacoplado; cada servicio invalida lo que le interesa |
| D-CFG-02 | Validacion de grafo de workflow | Solo al activar (draft permite grafo incompleto) | Permite iteracion; estabilidad al activar |
| D-CFG-03 | Almacenamiento de credenciales | Encriptacion AES-256 en DB | Simple para MVP; sin dependencia externa; migracion futura a vault posible |
| D-CAT-01 | Validacion consistencia familia-categoria-atributos | Frontend resuelve prefijo + backend valida | Doble validacion; UX fluida; funcion pura determinista |
| D-CAT-02 | Eliminacion de productos con solicitudes | Solo deprecar (no eliminar si tiene planes) | Datos siempre consistentes; respeta database-per-service |
| D-CAT-03 | Ownership planes de comisiones | Catalog Service owner, UI bajo Configuracion | Pragmatico; alineado con HU019 implementada |
| D-AUD-01 | Fuente de consulta para log de auditoria | PostgreSQL primario (ES solo logs operativos) | Fuente de verdad; sin dependencia extra; evolucion a ES cuando volumen lo justifique |
| D-ORG-01 | Eliminacion de organizaciones con entidades | Solo eliminar si no tiene entidades (409) | Sin cascada peligrosa; entidades son gateway a otros datos |
| D-ORG-02 | Eliminacion de entidades con sucursales | Solo desactivar si tiene sucursales; sin sucursales se elimina | Conservador; evita problemas cross-service |
| D-OPS-01 | Estado de solicitud vs workflow stage | Hibrido: status fijo (state machine) + workflow_stage libre (informativo) | Estabilidad en transiciones criticas; flexibilidad para BPM |
| D-OPS-02 | Concurrencia en liquidacion | Lock optimista con FOR UPDATE al confirmar | Verifica status=approved; 409 si cambio; sin lock pesimista |
| D-OPS-03 | Resolucion de variables de plantilla | Operations resuelve antes de publicar al bus | Operations tiene los datos; Notification solo despacha; desacoplado |
